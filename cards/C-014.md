status: todo

## Scope

Prototype real C Code execution for the documented `void _fn(int _number1) { ... }` block. This is security-class work: the result must be sandboxed, disabled by default unless safe, and must never execute arbitrary code in the browser main thread.

1. **Sandbox architecture** - choose and document the execution path, such as Web Worker + WASM runtime or a server-side sandbox with strict resource limits.
2. **API whitelist** - define the only simulator/hardware APIs C code may call. No DOM, localStorage, network, filesystem, or arbitrary JS access.
3. **Compilation/execution boundary** - support one minimal C function with numeric input/output and deterministic execution.
4. **Resource limits** - enforce timeout, memory limit where supported, output limit, and deterministic failure reporting.
5. **Interpreter bridge** - C Code block must generate an IR command/payload that invokes the sandbox adapter, not inline unsafe code.
6. **Error handling** - compile errors, runtime errors, timeout, and disallowed API usage produce diagnostics visible to the program/UI.
7. **Feature flag** - keep C execution off unless the sandbox passes verification; fallback is explicit `blocked-by-sandbox` diagnostic.
